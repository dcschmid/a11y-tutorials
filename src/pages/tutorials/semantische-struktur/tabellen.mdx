Benutzer von Bildschirmlesegeräten können in Tabellen fast wie in Tabellenkalkulationen navigieren, indem sie von einer Zelle zur
anderen in jede Richtung gehen. Wenn die Tabelle richtig beschriftet ist, liest das Bildschirmlesegerät zusätzlich zu den Daten in
der Tabellenzelle auch die Tabellenüberschrift vor, so dass blinde Nutzer/innen verstehen können, wo sie sich im Kontext der Tabelle
befinden.

Wenn Tabellen nicht richtig gekennzeichnet sind, werden Bildschirmleser wahrscheinlich Schwierigkeiten haben, die Datenbeziehungen
zu verstehen.

## Semantisches Markup für tabellarische Daten

### Tabellarische Daten sollten in einer `<table>` dargestellt werden.

Screenreader-Nutzer können in Datentabellen von Zelle zu Zelle navigieren, und zwar in mehreren Richtungen (nach oben, unten, rechts,
links), ähnlich wie in einer Tabellenkalkulation. Während sie sich von Zelle zu Zelle bewegen, lesen Screenreader die zugehörigen
Kopfzeilen (wenn die Tabelle unter Berücksichtigung der Barrierefreiheit erstellt wurde). Dank dieser Struktur und der einfachen
Navigation mit Bildschirmlesegeräten eignen sich Tabellen sehr gut für tabellarische Daten.

#### Erstelle keine falschen Tabellen

Die Verwendung von Tabellen hat einen Nachteil: Sie werden auf mobilen Geräten nicht immer besonders gut angezeigt. Dieser Nachteil
hat Entwickler dazu veranlasst, nach alternativen Möglichkeiten zu suchen, tabellarische Daten darzustellen. Vielen dieser Techniken
ist gemeinsam, dass sie `<div>`-Elemente und CSS verwenden, um das Aussehen von Tabellen nachzubilden, ohne die zugrunde liegende
semantische Struktur zu verwenden.

Das Problem dabei ist, dass Screenreader die semantische Bedeutung und Struktur für blinde Nutzer nicht mehr erkennen können. Auch
die Möglichkeit, in Tabellenzellen multidirektional zu navigieren, wird dadurch unterbrochen. Da sowohl die Semantik als auch die
Navigation unterbrochen sind, können die Nutzer/innen die gefälschte Tabelle nur der Reihe nach durchgehen - in der Reihenfolge, in
der die Elemente im DOM erscheinen.

Es gibt viele Techniken, mit denen Tabellen auf mobilen Geräten besser funktionieren, aber nur wenige davon sind ideal. Wenn du die
verschiedenen Ansätze auswertest, wähle nach Möglichkeit eine Technik, die die semantische Datenstruktur einer echten Tabelle
beibehält, zumindest in der Desktop-Ansicht. Manchmal ist es notwendig, in der mobilen Ansicht zu einem anderen Format zu wechseln
(z. B. eine Liste mit Beschriftungen).

## Beschriftung/Name der Tabelle

### Datentabellen sollten eine programmatisch zugewiesene Überschrift oder einen Namen haben.

Bildschirmlesegeräte lesen die Beschriftung oder den Namen der Tabelle, wenn die Nutzer zu der Tabelle navigieren, damit sie wissen,
worum es in der Tabelle geht. Das caption-Element ist die einfachste Möglichkeit, einer Tabelle einen Namen zu geben. Es ist auch
möglich, aria-label oder aria-labelledby zu verwenden.

Auf einer Seite mit mehreren Tabellen können Screenreader-Nutzer eine Liste aller Tabellen auf der Webseite aufrufen, in der die
Überschriften (z. B. "Ergebnisse des ersten Quartals") und die Tabellengröße (z. B. "3 Spalten, 4 Zeilen") angezeigt werden. Hier
ist eine Tabellenliste, die mit dem VoiceOver Screenreader angezeigt wird:

![Eine Tabellenliste, die mit dem VoiceOver Screenreader angezeigt wird](/images/semantische-struktur/table-captions-voiceover.png)

Hier ist die gleiche Liste von Tabellen, wie sie mit dem JAWS Screenreader angezeigt wird:

![Eine Tabellenliste, die mit dem JAWS Screenreader angezeigt wird](/images/semantische-struktur/table-captions-jaws.png)

Wenn du die Beschriftungen weglässt, sind alle Tabellen im obigen Beispiel ununterscheidbar, weil sie alle die gleiche Größe haben.

Selbst wenn die Tabellen unterschiedlich groß wären, wäre es schwer zu erkennen, worum es in den Tabellen geht, wenn sie keine
Beschriftungen haben.

#### Gutes Beispiel: Tabelle mit `<caption>` Element

Der Name/die Überschrift ("Ergebnisse des 1. Quartals") der unten stehenden Tabelle wird über das `<caption>`-Element programmatisch
mit der Tabelle verknüpft, sodass sie für Bildschirmleser verfügbar ist.

```html
<table>
  <caption>
    1st Quarter Results
  </caption>
  <thead>
    <tr>
      <td>&nbsp;</td>
      <th scope="col">Goal</th>
      <th scope="col">Actual</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">January</th>
      <td>$200,000</td>
      <td>$150,000</td>
    </tr>
    <tr>
      <th scope="row">February</th>
      <td>$200,000</td>
      <td>$240,000</td>
    </tr>
    <tr>
      <th scope="row">March</th>
      <td>$200,000</td>
      <td>$370,000</td>
    </tr>
  </tbody>
</table>
```

Eine alternative Methode mit aria-label (beachte, dass der Text im aria-label-Attribut für sehende Nutzer unsichtbar ist):

```html
<table aria-label="Second quarter Results"></table>
```

Diese Methode bietet ebenfalls eine unsichtbare Beschriftung, verwendet aber natives HTML und CSS statt ARIA:

Hinweis: Wenn du die Klasse visually-hidden direkt auf dem `<caption>`-Tag platzierst, liest der NVDA-Bildschirmleser die Tabelle
mit einer falschen Anzahl von Zeilen.

```html
<table>
  <caption>
    <span class="visually-hidden">1st Quarter Results</span>
  </caption>
</table>
```

Eine andere Methode, die aria-labelledby verwendet:

```html
<h3 id="tableCaption">Third Quarter Results</h3>
<table aria-labelledby="tableCaption"></table>
```

Eine Variante dieser Methode ist, den Namen in eine `<figcaption>` zu setzen und ihn über aria-labelledby mit der Tabelle zu
verknüpfen:

```html
<figure>
  <table aria-labelledby="figCaption">
    <!-- the rest of the table goes here -->
  </table>
  <figcaption id="figCaption">Fourth Quarter Results</figcaption>
</figure>
```

#### Schlechtes Beispiel: Tabelle mit falscher Beschriftung

Die Tabelle hat keine Überschrift, sondern eine Zeile, die sich über alle Spalten erstreckt und einen Text für die
Tabellenüberschrift enthält. Screenreader behandeln diese übergreifende Zelle nicht als den Namen der Tabelle, sodass sie in
Tabellenlisten nicht richtig angezeigt wird. Für Screenreader ist die Organisation der Tabelle aufgrund der übergreifenden Zelle
anfangs etwas schwer zu verstehen.

```html
<table>
  <tr>
    <th colspan="3">Revenue Goals</th>
  </tr>
  <tr>
    <td>&nbsp;</td>
    <th scope="col">Goal</th>
    <th scope="col">Actual</th>
  </tr>
  <tr>
    <th scope="row">January</th>
    <td>$200,000</td>
    <td>$150,000</td>
  </tr>
  <tr>
    <th scope="row">February</th>
    <td>$200,000</td>
    <td>$240,000</td>
  </tr>
  <tr>
    <th scope="row">March</th>
    <td>$200,000</td>
    <td>$370,000</td>
  </tr>
</table>
```

Ein ähnliches Problem tritt auf, wenn der Name der Tabelle nicht programmatisch mit der Tabelle verknüpft ist:

```html
<h3>Revenue Goals</h3>
<table>
  <!-- the rest of the table goes here -->
</table>
```

Der Text sieht für sehende Nutzer wie eine Tabellenüberschrift aus, wird aber von Screenreadern nicht als solche erkannt.

### Der Name/die Überschrift einer Datentabelle sollte die Identität oder den Zweck der Tabelle genau, aussagekräftig und prägnant beschreiben.

Der Zweck der Beschriftung oder des Namens ist es, dass die Nutzer die Tabelle identifizieren und/oder den Zweck der Tabelle verstehen
können, daher sollte die Beschriftung oder der Name sinnvoll sein. Einfach nur "Tabelle 1" zu sagen, wäre zum Beispiel nicht hilfreich.
Eine Alternative wäre "Tabelle 1: Ergebnisse des ersten Quartals".

### Der Name/die Überschrift jeder Datentabelle sollte im Zusammenhang mit anderen Tabellen auf derselben Seite eindeutig sein.

Wenn sich auf einer Seite zwei oder mehr Tabellen mit der gleichen Überschrift oder dem gleichen Namen befinden, können
Screenreader-Nutzer die Tabellen nicht unterscheiden, wenn sie die Liste aller Tabellen auf der Seite aufrufen.

## Tabellenüberschriften

### Tabellenüberschriften müssen mit `<th>` gekennzeichnet werden.

Screenreader können nur erkennen, ob eine Tabellenzelle eine Kopfzeile ist, wenn sie mit `<th>` markiert ist. Andernfalls gehen
Bildschirmleser davon aus, dass die Tabelle nur der visuellen Formatierung dient und lesen die Kopfzeileninformationen nicht wie
gewünscht.

Wenn sich die Kopfzellen in der ersten Zeile befinden, gehen Bildschirmleser davon aus, dass sie sich auf die darunter liegenden
Spalten beziehen mit anderen Worten, es gibt ein implizites scope="col". Es ist trotzdem eine gute Idee, den Bereich explizit zu
machen, indem du einen der folgenden Bereiche hinzufügst: col, row, colgroup oder rowgroup.

### Der Kopftext der Datentabelle muss die Kategorie der entsprechenden Datenzellen genau beschreiben.

Wenn der Text in der Kopfzeile ungenau ist (z. B. "Spalte 2") oder fremde Informationen enthält (z. B. Links, Schaltflächen oder
zusätzliche Beschreibungen, die nicht dem Namen der Spalte entsprechen), kann er für Bildschirmleser verwirrend sein. Am besten ist
es, den Text in der Kopfzeile genau und einfach zu halten.

einfache Zuordnungen stehen die Optionen scope="col" für Spaltenüberschriften und scope="row" für Zeilenüberschriften zur Verfügung.

## Gruppierte Kopfzeilenzuordnungen

### Die Überschriften der Datengruppen einer Tabelle müssen mit den entsprechenden Datenzellengruppen verknüpft werden.

Wenn du Tabellenkopfzellen zusammenführst, musst du eine Kopfzeile als für mehrere Zeilen oder Spalten entsprechend kennzeichnen.
Am einfachsten ist es, scope="rowgroup" oder scope="colgroup" zu verwenden. Screenreader lesen die mehreren Überschriften, wenn
sie durch die Tabellen navigieren.

Hinweis: Screenreader-Nutzer finden Tabellenkopfgruppen manchmal etwas verwirrend, selbst wenn sie perfekt markiert sind. Wenn
möglich, erstelle einfachere Tabellen, die keine verbundenen Zellen oder Kopfzeilengruppen benötigen.

## Komplexe Kopfzeilen-Assoziationen

### Header/Daten-Verknüpfungen, die nicht mit `<th>` und scope bezeichnet werden können, müssen mit headers plus id bezeichnet werden

Tabellen mit einer komplexen Struktur - in denen Datenzellen zusammengeführt werden oder es mehr als zwei Ebenen von Kopfzeilen für
eine bestimmte Dimension gibt - erfordern eine andere Technik als das einfache Markieren der Kopfzellen mit einem Bereich von col,
row, colgroup oder rowgroup. Jede Datenzelle muss mit Hilfe von headers und id explizit mit der entsprechenden Kopfzelle verknüpft
werden.

Warnung:
Komplexe Tabellenstrukturen mit unregelmäßigen Spalten- oder Zeilenbezeichnungen sind für Benutzer von Bildschirmlesegeräten schwer zu
verstehen und zu navigieren, selbst wenn das gesamte Markup "perfekt" für die Einhaltung der Barrierefreiheit ist. Es ist viel besser,
eine einfache Tabelle ohne übergreifende Zellen zu erstellen, als eine verwirrende, komplexe Tabelle, die technisch gesehen die
Zugänglichkeitsrichtlinien erfüllt.

Einige Screenreader, vor allem auf mobilen Geräten, unterstützen keine barrierefreien komplexen Tabellenauszeichnungen. Alte Versionen von
VoiceOver für OS X (bis Mac OS X 10.10.2) unterstützten ebenfalls keine komplexen Tabellen.

gestaltete Tabelle zu kompensieren.

## Tabelle Zusammenfassung

#### Für Datentabellen kann eine Zusammenfassung erstellt werden.

Eine Tabellenzusammenfassung ist nicht erforderlich. Im Idealfall ist die Tabelle so einfach und übersichtlich, dass sie sich allein
durch die Überschriften und die Datenstruktur von selbst erklärt. Trotzdem kann es manchmal hilfreich sein, eine kurze Zusammenfassung der
wichtigsten Punkte der Tabelle zu geben, um Datentrends, Datenformate und/oder andere Aspekte der Tabelle zu verstehen, die die Tabelle für
Screenreader-Nutzer verständlicher machen.

Zusammenfassungen sollten kurz und prägnant sein: in der Regel ein oder zwei Sätze, bei komplexeren Tabellen aber auch länger. Es gibt
verschiedene Möglichkeiten, Tabellenzusammenfassungen zu präsentieren:

- Füge die Zusammenfassung in den Absatz vor oder nach der Tabelle ein (mit aria-describedby)

- Füge die Zusammenfassung in die `<caption>` der Tabelle ein, solange die Zusammenfassung kurz ist.

- Setze die Tabelle in eine `<figure>` und füge den Titel und die Zusammenfassung in die `<figcaption>` ein.

- Schreibe die Zusammenfassung in das Attribut summary (für HTML-Versionen vor HTML5)

##### Gutes Beispiel: Methode 1: Füge die Zusammenfassung in den Absatz vor oder nach der Tabelle ein (mit aria-describedby)

## Layout-Tabellen

### Tabellen sollten nicht für ein rein visuelles (nicht datenbezogenes) Layout verwendet werden.

Von der Verwendung von Tabellen für das visuelle Layout und nicht für Datentabellen wird abgeraten, sie ist aber nach den
Zugänglichkeitsrichtlinien technisch zulässig. Die Verwendung von Layout-Tabellen hat keine schwerwiegenden Folgen für die
Barrierefreiheit, aber Tabellen wurden aus Sicht der semantischen Struktur für Datenraster entwickelt, daher ist es besser, sie nur
für diesen Zweck zu verwenden. Außerdem ist CSS eine robustere Methode, um visuelle Layouts für das moderne Web zu gestalten.

Wenn eine Layout-Tabelle verwendet werden muss, sollte dem `<table>`-Element role="presentation" hinzugefügt werden, um sicherzustellen,
dass Screenreader den Inhalt von Layout-Tabellen als Text und nicht als Tabelle lesen (und damit die semantische Tabellenauszeichnung
ignorieren) und sie wie div- oder `<p>`-Elemente behandeln.

### Layouttabellen müssen keine Datentabellenauszeichnung enthalten.

Wenn eine Tabelle ausschließlich für visuelle Layoutzwecke verwendet wird, solltest du keine der Kopfzeilenauszeichnungen für
Datentabellen einfügen, da Bildschirmlesegeräte die Benutzer darüber informieren, dass eine Tabelle vorhanden ist, und die Benutzer
erwarten, dass die Tabelle tabellarische Daten darstellt, was bei Layouttabellen nicht der Fall ist.

Insbesondere dürfen Layout-Tabellen nicht enthalten:

- Das `<caption>` Element

- Das Attribut `<summary>`

- Das `<th>`-Element

- Das scope-Attribut

- Das headers-Attribut

## Navigieren in Tabellen mit Screen Readern

### Wie Screenreader Tabellen lesen

Die Software des Bildschirmlesegeräts teilt dem Nutzer mit, dass es eine Tabelle gibt und wie viele Spalten und Zeilen sie hat, z. B.
"Tabelle mit 4 Spalten und 5 Zeilen".

Daraufhin kann der/die Nutzer/in entscheiden, ob er/sie den Bildschirmleser alle Zellen im normalen Modus durchlesen lassen will oder
ob er/sie Zelle für Zelle navigieren will (manchmal auch als "Tabellennavigationsmodus" bezeichnet). Im normalen Modus beginnt das
Lesegerät in der oberen linken Zelle der ersten Zeile und liest die gesamte Zeile vor, dann geht es zur nächsten Zeile und liest sie
von links nach rechts und so weiter. Im Tabellennavigationsmodus kann der Nutzer in jede beliebige Richtung navigieren, indem er die
entsprechende Tastenkombination im Bildschirmlesegerät verwendet.
